# Mio 聊天页精修与沉浸式语音通话设计

日期：2026-06-11

## 1. 设计目标

本次调整聚焦两个体验：

1. 普通聊天页减少装饰与常驻人物干扰，成为适合长期使用的低密度文字工作区。
2. 用户主动进入语音通话后，切换到独立的全屏沉浸空间，以右侧人物、底部字幕和清晰通话控制建立陪伴感。

视觉方向采用“暖梅精修”：

- 保留 Mio 现有暖灰、米白、暗梅品牌色。
- 不改为常见蓝紫 AI 主色。
- 减少约 40% 的阴影、渐变、玻璃和无意义悬浮动效。
- 通过留白、排版、实体表面和少量空间光建立专业感。

## 2. 产品结构

```text
普通聊天页
  -> 用户点击语音入口
  -> 进入全屏通话壳并请求麦克风授权
  -> 授权成功后连接语音服务
  -> ASR / Agent / TTS / 字幕 / 人物表现
  -> 用户结束或退出
  -> 返回原聊天页和原 conversation_id
```

文字与语音仍共享同一个 Conversation、Persona、Memory、RAG、Safety 和 Agent Trace。全屏通话只是独立的 Experience Layer，不创建第二套人格、记忆或聊天历史。

## 3. 普通聊天页

### 3.1 删除内容

- 删除顶栏“显示澪”开关。
- 删除聊天页常驻 `AvatarStage`。
- 删除人物旁“收起澪”按钮。
- 删除消息区和输入区为人物预留的右侧安全区。
- 删除与常驻人物有关的显示偏好。

### 3.2 保留内容

- 左侧导航、会话列表和新对话入口。
- 顶栏服务状态与当前会话信息。
- 消息历史、流式回复、失败重试和停止生成。
- 底部输入框、附件入口和主动语音入口。

### 3.3 视觉精修

- 聊天内容恢复完整居中宽度，正文成为首要信息焦点。
- 顶栏减少胶囊控件数量，语音入口作为明确的次级操作。
- 普通表面优先使用实体暖白，磨砂只用于输入框和必要浮层。
- 普通消息不使用扫光动画；悬停反馈仅保留轻微颜色或阴影变化。
- 侧栏降低选中态、开发中徽标和按钮阴影的视觉重量。
- 暗梅用于主操作、用户消息和焦点，不扩大为大面积背景。

## 4. 全屏语音通话

### 4.1 页面构图

- 通话页覆盖整个应用工作区，不保留左侧导航和聊天顶栏。
- 背景使用暖灰到米白的低对比渐变，并带低饱和灰粉环境光。
- 人物位于右侧前景，高度约为桌面可视区域的 `72%–86%`。
- 左侧保留大面积留白，可显示通话状态、简短陪伴文案和音量反馈。
- 字幕位于人物下方偏中区域，不贴底，不与控制按钮重叠。
- 控制区固定在页面底部中央。

### 4.2 顶部信息

- 左上为明确的关闭按钮，触控区域至少 `44 × 44px`。
- 顶部中央显示“与澪通话中”和当前阶段。
- 右上显示通话时长。
- 关闭按钮与底部“结束”均进入同一结束流程。

### 4.3 字幕

- 默认展示澪当前实际播放或即将播放的句子。
- 用户说话时可切换显示临时转录，并明确标注说话者。
- 字幕使用半透明暖白表面、暗色高对比文字和轻量模糊。
- 桌面端最大宽度约为页面的 `56%`，移动端宽度约为 `90%`。
- 长文本按句切换，不把完整长回复塞进单个字幕块。
- 用户打断后只把实际播放过的内容记录为 `heard_text`。

### 4.4 底部控制

- 静音：只控制麦克风输入，不结束通话。
- 结束：使用暗梅主按钮，停止录音、播放和媒体会话。
- 字幕：允许显示或隐藏字幕，但默认开启。
- 后续实时通话可增加扬声器、设备和重连控制，不在首版堆叠。

### 4.5 状态

```text
requesting_permission
  -> connecting
  -> listening
  -> transcribing
  -> thinking
  -> speaking
  -> interrupted
  -> reconnecting
  -> ended / failed
```

- `requesting_permission`：全屏通话壳解释麦克风用途；拒绝后显示返回聊天页的明确操作。
- `listening`：左侧显示低幅 Voice Teal 音量反馈。
- `transcribing`：字幕显示用户临时转录。
- `thinking`：人物保持克制待机，显示简短状态。
- `speaking`：字幕、音频、嘴型和表现事件同步。
- `interrupted`：立即停止未播放片段，恢复监听。
- `reconnecting`：保留页面与已生成字幕，提供重试或结束。
- `failed`：解释 ASR、TTS、人物或网络故障，并提供降级路径。

## 5. 降级与隐私

人物资源降级顺序：

```text
当前配置的主渲染器（VRM 或 Live2D）
  -> 静态透明立绘
  -> 圆形头像和环境光
  -> 无人物的纯语音与字幕
```

- 人物失败不得阻断语音或文字对话。
- TTS 失败时保留字幕，并允许返回聊天页阅读完整回复。
- ASR 失败时允许重试或结束通话，不伪造用户消息。
- 默认不长期保存原始音频。
- 首次录音必须由用户主动触发并授权。

## 6. 响应式

### 桌面

- 左侧信息约占 `35%–42%`，右侧人物约占 `40%–48%`。
- 字幕避开人物脸部、嘴型和底部控制。

### 移动端

- 人物缩小并下沉到右侧，约占屏幕高度的 `58%–70%`。
- 左侧陪伴文案简化，只保留通话状态和音量反馈。
- 字幕置于底部控制上方，宽度约 `90%`。
- 通话时长可在窄屏隐藏，状态信息不得被遮挡。

## 7. 组件边界

- `ChatPage`：普通文字聊天，不处理人物渲染。
- `VoiceCallPage`：全屏通话布局和状态组合。
- `VoiceCallControls`：静音、结束和字幕开关。
- `VoiceSubtitle`：说话者、临时转录和播放字幕。
- `AvatarStage`：仅在 `VoiceCallPage` 中渲染人物和降级资源。
- `VoiceSession`：管理授权、媒体状态、取消和重连，不维护 Persona。
- `PresentationEngine`：把抽象语义映射到人物、嘴型和声音表现。
- `AvatarController`：管理 AvatarProfile、Renderer 生命周期、事件顺序和降级。
- `AvatarRenderer`：可插拔实现 Static、Live2D 和 VRM，不读取 Conversation 或 Memory。

## 8. 首版实现边界

首版 UI 可以使用 Mock Voice Session 验证：

- 从聊天页进入和退出全屏通话。
- `listening`、`thinking`、`speaking`、`failed` 状态。
- 静音、结束和字幕开关。
- 静态人物与无人物降级。
- 返回聊天页后保持原会话。

首版不要求立即实现 WebRTC、VAD、真实 ASR/TTS、可打断实时媒体或设备切换。

VRM/VRMA 不属于首版 UI 的前置条件。先使用静态人物完成通话闭环，再按 `2026-06-11-vrm-avatar-renderer-design.md` 引入统一 Renderer 抽象和 `VrmAvatarRenderer`。

## 9. 验收标准

- 普通聊天页不再出现人物、人物开关或人物安全区。
- 只有用户点击语音入口后才申请麦克风，不在页面加载或普通聊天时申请。
- 通话页全屏显示，不保留应用导航。
- 人物在右侧，字幕在人物下方且不与控制区重叠。
- 结束后返回原聊天和原滚动上下文。
- 人物、TTS 或 ASR 失败时存在明确降级。
- 桌面和 `390 × 844` 移动视口均无横向滚动。
- 所有关键状态记录 VoiceTrace，并复用现有 conversation_id。

## 10. 设计决策

- 选择“暖色静室”，不选择深色影院或项目桌面背景。
- 主题色不推翻，只降低装饰浓度并收紧色彩角色。
- 人物从普通聊天页移除，只在用户主动进入语音通话时出现。
- 独立全屏页面不代表独立 Agent；它只隔离交互与媒体生命周期。
