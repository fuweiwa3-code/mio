# Mio UI Brief

## 产品身份

Mio 是面向程序员的陪伴型 AI，而不是披着角色外观的开发工具。澪首先要让用户感到她持续在场、理解情绪并记得关系，其次才提供项目上下文、知识检索和调试协助。

## 核心体验

- 默认入口是文字聊天。
- 普通聊天页不常驻展示人物，保持完整文字内容宽度。
- 用户主动点击语音入口后进入全屏沉浸式通话。
- 通话页人物位于右侧，字幕位于人物下方，控制区位于底部。
- 进入通话前必须由用户主动授权和开启麦克风。
- 结束通话后返回原聊天和原会话上下文。
- 当前配置的 Live2D 或 VRM 主渲染器加载失败时，降级为静态立绘、头像或无人物模式，不影响语音和字幕。

## 页面优先级

1. 聊天与陪伴。
2. Persona Builder 与 Persona Settings。
3. 长期记忆、知识库和项目上下文。
4. Agent Trace 与 Debug Console。
5. 主动关怀、提醒与表情包。

## 信息密度

- 日常聊天保持低密度和充分留白。
- 项目上下文、记忆管理和 Debug Console 使用更紧凑的桌面工作台结构。
- 功能页不能覆盖陪伴身份，应保留澪的状态、语气和返回聊天入口。

## 目标设备

- 桌面 Web 是 V1 主体验。
- 移动端保持完整聊天能力。
- 移动端聊天保持完整单列；全屏通话中的人物缩小并贴近右侧，字幕和控制不得被遮挡。
- 所有触控目标至少为 44px。

## 已确认的视觉方向

采用“A 为主、C 为辅”的温暖融合方案：

- **清冷暖色日常**：聊天页使用暖灰、米白、暗梅色，舒适、克制且适合长期使用。
- **轻盈桌面结构**：项目上下文、记忆管理和 Debug Console 使用更清晰紧凑的工作台布局。
- **局部磨砂与空间层次**：仅用于输入框、人物控制和浮层，不使用满屏玻璃。
- **通话人物焦点**：人物只在全屏语音通话中通过软阴影、轻微景深和暖色环境光建立在场感。
- **高对比内容层**：聊天文字、代码和 Trace 必须清晰易读。
- 所有页面共享颜色、字体、组件和交互反馈，不形成割裂产品。
